Reservationen

Web-Applikation für Reservationen

Mit der Web-Applikation für Reservationen
- kann der Konzertbesucher selbst Plätze auswählen
- wird eine Reservationsbestätigung automatisch gesendet
- kann am Konzerttag Reservationskarten gedruckt werden

Reservation auf der Homepage zufügen

Konzertbescher reservieren Plätze auf der JAZZ live AARAU Homepage. In der App sowie
im Newsletter gibt es ein Link zur Homepage Webeseite.

Das Homepage Reservationsfenster öffnet die Webseite MakeReservation.htm um Plätze
auszuwählen und zu speichern.

Nach dem Speichern der Reservationsdaten (XML Datei), wird eine Reservations-Bestätigung an
die im Homepage Reservationsfenster eingegebene E-Mail-Adresse gesendet.

Eine Kopie der E-Mail wird (als BCC) auch an reservation@jazzliveaarau.ch geschickt.

Falls die eingegebene E-Mail-Adresse ungültig ist, wird eine E-Mail mit Information darüber
vom Server an reservation@jazzliveaarau.ch gesendet.

Die Weiterleitungsadresse für die Fehler-E-Mail kann man ändern. Instruktionen hier als PDF und als DOCX


Im Intranet Reservationen zufügen und administrieren

Mit der Webseite AddReservation.htm können Reservationen erfasst, gespeichert sowie eine Liste
mit allen Reservationen generiert werden.

Vor dem Konzert können Reservationskarten mit einer Funktion in AddReservation.htm gedruckt werden.


Multi-Anwender Lösung

Nach jeder Platzwahl und vor dem Speichern kontrolliert die Applikation ob durch jemand anders (gleichzeitig)
Plätze reserviert wurden. Falls dies zutrifft, wird der Benutzer die Meldung erhalten, neue Plätze
gewählt werden müssen. Die von ihm gewählten Plätze wechseln die Farbe von gelb zu rot.

Da die Anzahl Reservationen nicht gross ist, ist dieser Fall eher unwahrscheinlich.

Eine Lösung mit Checkout, wie für die Applikationen Admin und Adressen, ist also nicht implementiert.


Hilfe Dokument für MakeReservation.htm

Es gibt ein Dokument, dass das Speichern von der XML Reservations-Datei und die Multi-Anwender Lösung
in detail (für einen Entwickler) beschreibt: Als PDF sowie DOCX


Maximale Anzahl reservierte Sitzplätze

In der XML Datei für den Tischplan (LayoutSalmen.xml) ist auch die maximale Anzahl reservierte Sitzplätze.
Es ist eine Prozentzahl von der Anzahl Sitzplätze. Normaler Wert is 50%;


Reservationsbestätigung

Für E-Mails an reservation@jazzliveaarau.ch erhält man eine automatische Antwort.

Die von MakeReservation.htm automatisch geschickte Bestätigungs-E-Mails enthält
nachfolgenden Text, zudem auch die detaillierten Reservationsdaten.

----------------------------------------------------------------------

Liebe Konzertbesucherin, lieber Konzertbesucher

Die erfolgte Reservation ist bei uns eingetroffen, besten Dank.
Die gewünschte Anzahl Plätze wird für Sie bereitgestellt und muss
bis spätestens 10 Minuten vor Konzertbeginn eingenommen werden.

Wir wünschen Ihnen bereits jetzt ein unvergessliches Konzerterlebnis.

Herzlich
JAZZ live AARAU

----------------------------------------------------------------------

Browserdaten löschen

Jeder Browser speichert Webseiten auf dem Computer. Beim Surfen, wenn man zurück zu einer Seite geht, wird sie nicht wieder
vom Server heruntergeladen. Sie wird vom Computer geholt. Das nennt man Browser-Cache und macht das Surfen viel schneller.
Der Nachteil ist, dass die im Cache gespeicherten Daten veraltet sein können, wenn die Webseite zwischenzeitlich aktualisiert wurde.

Beim Reservieren öffnet der Benutzer die Eingabe-Webseite StartReservation.htm (für Name, E-Mail und Bemerkung) zuerst.
Danach Webseite MakeReservation.htm wo die Plätze selektiert werden. Nach Speichern der Reservation wird MakeReservation.htm
zugemacht, und der Benutzer kommt wieder zur Webseite StartReservation.htm.

Wenn der Benutzer noch eine Reservation machen möchte, soll die Webseite MakeReservation.htm nicht vom eigenen Computer geholt werden,
sondern wieder vom Server heruntergeladen werden, d.h. der Benutzer sollte vorher die Browserdaten löschen. Das ist aber nicht notwendig,
weil die Webseite MakeReservation.htm instruiert (mit so genannten meta-daten) den Browser diese Seite immer vom Server herunterzuladen.
Wie man das macht ist hier beschrieben.

Leider muss der Entwickler doch die Browserdaten löschen, nach Änderungen in den JavaScripts, die in separaten Dateien auf dem Server sind.
Die Webseite wird wieder heruntergeladen, aber nicht die zugehörigen JavaScript-Dateien.

Für die Homepage könnte man auch versuchen solche meta-daten einzuführen. Leider funktioniert es vermutlich nicht für alle Änderungen,
die Programm Admin macht. Beim Aktualisierung der Homepage wird eine Daten-JavaScript-Datei kreiert und auf den Server hochgeladen.
Diese Datei wird vermutlich nicht wieder heruntergeladen. Vielleicht kann man den Browser auch instruieren JavaScript-Dateien immer neu zu laden.
Muss untersucht werden...
